# API

Full interactive reference (request/response schemas, try-it-out) is auto-generated by FastAPI and
served at **`/docs`** (Swagger UI) and **`/redoc`** whenever the API is running, plus the raw schema
at `/openapi.json`. This document is a map of what exists and how auth/pagination work — treat
`/docs` as the source of truth for exact payloads.

Base path: `/api/v1`. Realtime events: `GET /api/v1/ws` (WebSocket).

## Authentication

Two schemes, both sent as `Authorization: Bearer <token>`:

1. **JWT access tokens** — from `POST /auth/register` or `POST /auth/login`. Short-lived
   (`access_token_expire_minutes`, default 30 min); refresh via `POST /auth/refresh` with the
   longer-lived `refresh_token` (default 7 days).
2. **API keys** — created with `POST /auth/api-keys` (shown once, prefixed `djs_`), optionally
   scoped to a single project. Pass the raw key as the bearer token; the API detects the `djs_`
   prefix and looks it up by SHA-256 hash.

Endpoints under `/projects/{project_id}/...` additionally check organization membership (JWT) or
project scope (API key) via the `require_project_access` dependency.

## Endpoints by resource

| Resource | Endpoints |
|---|---|
| Auth | `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh`, `GET /auth/me`, `POST /auth/api-keys` |
| Organizations | `GET /organizations`, `GET/POST /organizations/{id}/projects`, `PATCH /organizations/{id}/projects/{project_id}` |
| Queues | `GET/POST /projects/{id}/queues`, `PATCH /projects/{id}/queues/{queue_id}`, `POST .../pause`, `POST .../resume`, `GET .../metrics` |
| Retry policies | `GET/POST /projects/{id}/retry-policies` |
| Jobs | `POST /projects/{id}/jobs`, `POST .../batch`, `GET /projects/{id}/jobs` (paginated/filtered/searched), `GET/POST .../{job_id}`, `.../cancel`, `.../replay`, `.../executions`, `.../logs` |
| Dead letter queue | `GET /projects/{id}/dead-letter-queue?queue_id=...` (paginated) |
| Workers | `POST/GET /projects/{id}/workers`, `POST .../{worker_id}/heartbeat`, `GET .../heartbeats`, `POST .../shutdown` |
| Dashboard | `GET /projects/{id}/dashboard/overview` |
| Ops | `GET /health` (DB + Redis check), `GET /metrics` (Prometheus exposition format) |

## Pagination, filtering, sorting

List endpoints that return many rows (`jobs`, `dead-letter-queue`) return a `Page<T>` envelope:

```json
{ "items": [...], "total": 137, "page": 1, "page_size": 20, "pages": 7 }
```

Query params: `page`, `page_size` (max 200), and for jobs specifically `status`, `queue_id`, `search`
(substring match on job name), `sort_by`, `sort_order` (`asc`/`desc`).

## Creating a job

`POST /projects/{id}/jobs` — exactly one of `run_at`, `delay_seconds`, `cron_expression` may be set;
omitting all three runs the job immediately.

```json
{
  "name": "send-welcome-email",
  "queue_id": "…",
  "payload": { "task": "http_request", "args": { "url": "https://…" } },
  "priority": 10,
  "max_attempts": 5,
  "delay_seconds": 30
}
```

`payload.task` selects a handler from the built-in registry (`app/execution/registry.py`):
`echo`, `sleep`, `http_request`, `fail`, `random_fail`. Registering a new task is a one-line
`@task("name")`-decorated async function — see that file for the pattern.

## Errors

Non-2xx responses are `{"error": "message"}` (validation failures additionally include a `details`
array of Pydantic error objects). HTTP status codes follow convention: 401 (missing/invalid
credentials), 403 (authenticated but not authorized for this org/project/role), 404, 409 (conflicts —
duplicate queue name, terminal-state job cancellation), 422 (validation), 429 (rate limit).
